# Decision Record: Rendering Contract (V1)

## Status
Proposed (recommended default below).

## Context
We want to turn a user prompt into a diagram that can be:
- previewed as HTML (interactive)
- exported reliably as PNG/JPG
- stored as prompt(s) + generated JS/code in DB

The highest-risk area is executing AI-generated code safely and reliably.

## Options
### Option A: Direct JavaScript generation (D3/C3/etc.)
The model outputs JavaScript that directly renders the diagram.

**Pros**
- Matches the user-facing story (“generate JS”).
- Potentially more flexible visuals early.

**Cons**
- Security risk: arbitrary code execution vectors.
- Harder validation; requires AST allowlisting, sandboxing, and strict runtime constraints.
- Preview/export drift risk (if runtime differs).
- Harder to test deterministically.

### Option B: Structured DiagramSpec (JSON/DSL) + deterministic renderer (recommended)
The model outputs a constrained **DiagramSpec** (JSON) or small DSL.
The app compiles/renders it using a deterministic, maintained renderer.
Optionally, the system can still generate and store “JS code” by compiling the spec to JS.

**Pros**
- Safer: validate with JSON Schema; reject unsupported constructs.
- More deterministic: stable exports, easier snapshot tests.
- Easier to version: `specVersion` and renderer versioning.
- Better compatibility for headless export pipelines.

**Cons**
- Up-front work: define spec + renderer.
- Some diagram types may be harder until spec grows.

## Decision (V1)
Choose **Option B**: **DiagramSpec (JSON) -> deterministic renderer -> preview + export**.

We will still satisfy the requirement to store “generated JS code” by:
- storing the **DiagramSpec** as the primary artifact, and
- storing the **compiled JS bundle/snippet** that renders that spec (generated by our system, not arbitrary LLM code).

## DiagramSpec (V1) – minimal shape
V1 should support a general-purpose “scene graph” with primitives:
- canvas size: width/height, background
- primitives: line, polyline, polygon, circle, arc, rect
- text labels: text, position, font options, rotation
- styling: stroke, fill, strokeWidth, dash
- layers/zIndex

Optional extensions (not required in first thin slice):
- arrows/markers
- grids/axes
- math labels (LaTeX via KaTeX) with explicit enable flag

## Validation rules (V1)
- Spec must conform to schema (`specVersion: 1`).
- Hard caps: max primitives, max text length, max canvas size.
- Disallow external URLs/assets in spec.

## Export implications
If we render via SVG in preview, export can be:
- SVG -> PNG/JPG (server-side) OR
- headless Chromium rendering the same preview HTML

## Migration path to “advanced JS mode”
Later (post-MVP), we can add an opt-in “advanced JS” mode with:
- strict library/runtime allowlist
- AST validation (no `eval`, no dynamic imports, no network)
- sandbox isolation and timeouts
- separated trust tiers (only for authenticated/paid users, or internal)
